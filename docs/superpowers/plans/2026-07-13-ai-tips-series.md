## 0. 最初に必ずやること（各セッション共通）

1. 設計書を全文読む。
2. 既存記事を最低2本読み、文体・パーツの使い方を体に入れる:
   - `src/content/ai-tips/2026-07-09-clipboard-history.md`（qaふきだし・kbd・mockupの使い方の見本）
   - `src/content/ai-tips/2026-07-12-notebooklm-custom-instructions.md`（callout・prompt-share・注意事項セクションの見本）
3. `src/pages/ai-tips/[...slug].astro` を読み、利用可能なCSSパーツを確認する。
4. `HANDOVER.md` があれば直近の申し送りを確認する。
5. 途中セッションから再開する場合は、本書 §5 の進捗表を確認し、該当フェーズから着手する。

## 1. 進め方の大原則

- **1フェーズ＝1確認**。各フェーズの成果物ができたら必ず停止し、河原さんのレビューを受けてから次へ進む。まとめて先行しない。
- 記事は**必ず `draft: true` で作成**する。`draft: false` への変更は河原さんの明示的な承認後のみ（§4 公開チェックリスト）。
- 設計書 §5（編集方針）は逐語的に守る。特に:
  - フェアネスの原則（Geminiを貶めない・条件の違いだと毎回明示）
  - 内部アセット名は callout「nestでの実装例」内のみ（`ProviderFeedback` 等のコード識別子は出さない）
  - 個人情報・実データのマスキング
- 制度用語（加算名等）は設計書 §3 の素材の範囲を超えない。超える必要があるときは執筆せず、確認事項として提示する。
- 既存ファイルの変更は最小限に。`[...slug].astro` へのCSS追加以外に既存ページ・コンポーネント・スタイルを変更する必要が生じたら、事前に河原さんへ確認する。
- コミットは河原さんの運用に従う（指示がなければフェーズ完了時にまとめて。`draft: true` のままならコミット・pushしても公開はされないが、公開リポジトリかどうかを意識すること）。

## 2. フェーズ分割

### Phase 0｜共通基盤（記事より先に1回だけ）

**作業**
1. `src/pages/ai-tips/[...slug].astro` に設計書 §6.1 の2パーツを追加:
   - (a) qa話者バリエーション: `.qa__row--gemini` / `.qa__row--claude` ＋ 名札 `.qa__name`
   - (b) AI回答引用ボックス: `.ai-answer`（`--gemini` / `--claude` modifier、ヘッダに AI名＋条件ラベル）
2. 連載ナビの実装方式を決める（設計書 §6.3）。md内HTML直書きを推奨（Content Collections の制約のため）。雛形HTMLを作り、レビューに含める。
3. 見本表示用に、ダミー記事（`draft: true`）または既存ドラフト内で新パーツを使ったサンプルを用意し、`npm run dev` でスクリーンショットを撮る。

**DoD（完了条件）**
- `npm run build` が通る。
- 既存8記事の表示に影響がない（少なくとも2記事を目視確認）。
- 新パーツ2種＋連載ナビのスクリーンショットを河原さんに提示し、デザイン承認を得る。

### Phase 1〜5｜各記事の執筆（1記事＝1フェーズ）

対象と仕様は設計書 §4 の各回仕様に従う。**執筆順は第1回→第5回の順**。

各フェーズの共通手順:
1. 設計書 §4 の該当回仕様と §3.4 の素材配分を再読する。
2. 構成骨子（見出しレベルまで）を先に作り、**本文を書く前に河原さんへ骨子を提示**する（往復を減らすため、骨子と初稿を同時に出してもよいが、骨子の変更可能性が高い回＝第1回・第5回は骨子先行を推奨）。
3. 本文を執筆（`draft: true`、仮日付でファイル作成。分量目安 8〜15KB）。
4. 図解挿絵は**制作済み**（設計書 §6.4 の一覧参照。`public/images/ai-tips/` に配置済み）。記事には該当する挿絵をそのまま埋め込む。**実際の画面スクリーンショットが必要な箇所のみ**、プレースホルダ＋alt文＋撮影指示（何をどう撮るか）を入れた状態で提出し、撮影は河原さんの指示を待つ。
5. セルフチェック（§3）を実施。
6. `npm run dev` で表示確認し、記事全体のスクリーンショットを撮って河原さんに提出。**確認事項（設計書 §7 の該当項目＋執筆中に生じた判断）を箇条書きで添える。**

各回の固有注意:
- **第1回**: フェアネスcallout（設計書 §4 第1回の骨子文言）を冒頭に必ず配置。Gemini/Claude の回答引用は `.ai-answer` を使用し、原文の事実関係を変えない（要旨化は可）。連載予告に未公開回のリンクを張らない。
- **第2回**: MCPの説明は「手足」のたとえで統一。技術的に厳密な定義に踏み込みすぎない。「身近な第一歩」の題材は着手前に河原さんへ確認（設計書 §7-2）。
- **第3回**: 締めの問い（「道具だけでは、これは分かりません」）を必ず入れる。連載の背骨（設計書 §2）。Phase 0〜5 の工程図の形式（SVG図解か表か）は骨子段階で提案する。
- **第4回**: 「気をつけたいこと」callout（個人情報の扱い）は必須。書きぶりは設計書 §4 第4回-4 の範囲内で、実装の詳細（DB名・クエリ等）には踏み込まない。
- **第5回**: デモ出力の実データ掲載可否を**執筆前に**河原さんへ確認（設計書 §7-4）。既存記事（2026-07-12 カスタム指示）への内部リンクを入れる。**「親なき後」ツール・しくみページ（`/post-parent/tools/`）への接続を必ず入れる**（設計書 §4 第5回-4。台帳＝くらしサポート、関係図＝支援エコマップの対応で回収する）。掲載名・説明文は `src/lib/post-parent.ts` の現行表記に合わせ、リンク先の存在をビルドで確認する。お問い合わせ誘導の定型文で締める。

**各フェーズのDoD**
- セルフチェック（§3）全項目クリア。
- `npm run build` が通る。
- スクリーンショット＋確認事項リストを河原さんに提出済み。
- 河原さんのフィードバック反映が完了し、「この回はOK」の言質を得ている（公開＝`draft: false` はまだしない）。

### Phase 6｜公開（回ごとに繰り返し）

## 3. セルフチェックリスト（各記事共通）

執筆後、提出前に全項目を確認する。

- [ ] frontmatter: title / date / summary が揃い、`draft: true` である
- [ ] ファイル名が `YYYY-MM-DD-<スラッグ>.md` 形式（スラッグは設計書 §2 の案に従う）
- [ ] 冒頭に「中級者編」の明示と連載ナビがある
- [ ] フェアネス: Geminiを貶める表現・優劣断定がない。「条件の違い」であることが本文から読み取れる
- [ ] 内部アセット名が本文に漏れていない（callout「nestでの実装例」内のみ）。コード識別子（バッククォート付きの変数名等）が出ていない
- [ ] 個人名・利用者情報・実在事業所への評価と読める表現がない
- [ ] 制度用語の使い分け: 「強度行動障害支援体制加算」と「重度障害者支援加算」を混同していない（設計書 §5.4）
- [ ] 専門用語の初出に言い換え・説明がある
- [ ] 画像タグに width / height / alt / loading="lazy" / decoding="async" がある（プレースホルダ段階でも alt と図版指示は書く）
- [ ] 内部リンクが正しい（既存記事・/contact/ 等。未公開回へのリンクがない）
- [ ] 次回への「引き」が設計書 §2 のとおりに入っている
- [ ] 文体が既存記事と揃っている（ですます調・煽りなし・太字は要点のみ）
- [ ] `npm run build` が成功する

## 6. 判断に迷ったら

- 設計書に書いていない編集判断（表現・構成の変更、素材にない事実の追加、新しいCSSパーツの追加）は、**自分で決めずに確認事項として提出物に添える**。
- 「もっと良い構成を思いついた」場合も、まず現行仕様どおりに作るか、着手前に代案として提示するかのどちらかにする。黙って仕様から外れない。
- 素材（設計書 §3）の事実関係と矛盾する情報をWebで見つけた場合は、執筆に反映せず報告する。
