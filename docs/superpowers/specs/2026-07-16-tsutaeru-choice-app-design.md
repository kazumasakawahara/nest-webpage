# 設計書: 選択・気持ち伝達支援PWA「つたえるカード（仮）」

- 作成日: 2026-07-16
- 発注者: 河原さん（レビュー・承認担当）
- 設計: Claude Code（ブレインストーミングにて3セクション承認済み）
- 実施体制: Claude（指揮者・レビュー）＋ Opus 4.8 サブエージェント（実装担当）
- 対象リポジトリ: nest-webpage（nponest.com に同居）

## 1. 目的と背景

知的障害のある利用者さんが、**その場で選択肢を指差して意思や気持ちを伝えられる**スマホアプリを作り、nponest.com から「ホーム画面に追加」で配布する。

厚労省「障害福祉サービス等の提供に係る意思決定支援ガイドライン」の枠組みでいう**意思形成支援（分かる形で選択肢を見せる）と意思表明支援（指させる）**を道具にするもの。nest の意思決定支援ページの理念「知らないから伝えられない、を断ち切る」の実装でもある。

### スコープ（2026-07-16 河原さん確定）

- 主な利用端末: **支援者のスマホ**
- 対象場面: **B 気持ち・体調をつたえる／C 確認・合意／D ふりかえり・対話の補助／E17 通院での症状伝達**（日常の「どれにする」系はプリセットの柱から外す。カスタムテーマで対応可能）
- 記録: **端末内履歴＋テキスト書き出し**

## 2. テーマ構成（v1 プリセット7本）

アプリの中身は「テーマ（質問）」の集まり。1テーマ＝1〜3問の流れ、各問い2〜6枚の大カード。

| テーマ | 問いの流れ | カード例 |
|---|---|---|
| きもち | 気持ち → つよさ | うれしい・たのしい・おこってる・かなしい・こわい・つかれた → すこし／とても |
| こまった | 理由（1問） | うるさい・まぶしい・あつい/さむい・人がおおい・おなかすいた・ねむい・やることがわからない・まつのがいや |
| いやだった | 理由（1問） | おと・におい・あじ・はじめて・じゅんばんがちがう・まえにいやなことがあった |
| はい・いいえ | 確認（1問） | はい・いいえ・わからない・どっちもいや |
| おねがい | 要求（1問） | てつだって・やすみたい・トイレ・おみず・しずかなところにいきたい・やめたい |
| ふりかえり | きょうは？ → なにが？ → 分岐 | 下記詳細 |
| からだ（通院用） | どんないたみ？ → いつから → つよさ | 下記詳細 |

### ふりかえり（3問＋分岐）

1. きょうは？: たのしかった・ふつう・いやだった
2. なにが？: しごと・ごはん・おやつ・おでかけ・ひと・**からだのこと**（このセットは事業所の活動に合わせて支援者が差し替える前提のデフォルト）
3. 分岐: 「からだのこと」→ からだのようすカード（ねむれなかった・あたまがいたい・おなかがいたい・つかれている・ムカムカする 等）。「いやだった」の深掘りは既存「いやだった」テーマへワンタップ接続

問2に「からだのこと」を常設することで、**体調が背景にあるかもという問題意識を仕組みとして残す**（河原さん要望・2026-07-16）。

### からだ（通院用）: オノマトペカード

「どこが痛いか」は身振りで伝えやすいとの現場判断（河原さん）により、デフォルトの流れから「どこが」を外し（テーマ設定で復活可）、核を**痛み・感覚の質**に置く。

擬音語（オノマトペ）＋感覚のたとえ絵のカード8枚（v1仮説）:

| カード | たとえ絵 |
|---|---|
| ズキズキ | 脈打つ波 |
| キリキリ | ぞうきんをぎゅっとねじる |
| チクチク | 針・とげ |
| ガンガン | ハンマーでたたかれる |
| ヒリヒリ | 炎・ほてり |
| ドーンとおもい | おもい石をのせられた感じ |
| ムカムカ | 胃のうずまき |
| うまくいえない・なんとなくへん | もや・きり |

採用理由: ①診察室で医師にそのまま通じる語彙（本人の指差しが翻訳なしで医療の言葉になる）②読み上げ音のリズム自体が実感と照合しやすい。**この8枚は出発点であり、現場で試して本人に通じる札だけ残す「育てる」前提**。「うまくいえない」も診察上意味のある情報として扱う。

## 3. 画面の流れと共通原則

ホーム（テーマ大ボタン一覧）→ 支援者がテーマを選んで差し出す → 本人がカードをタップ → 大きく表示され**もう一度タップで決定**（誤タップ対策の2段階）→ 次の問い → 結果画面（選んだカードを並べて大表示。通院ではこの画面を医師に見せる）。

全テーマ共通:

- 逃げ道カード**「どれもちがう」「わからない」**を常設（設定でOFF可）。誘導を避ける意思決定支援の倫理の実装
- 並び順**シャッフル**オプション(いつも先頭を押す方への配慮)
- タップで**読み上げ**（端末内蔵音声合成。OFF可）
- **本人モード**: 差し出す前にワンタップでON。ホーム・設定への導線が消え、解除は画面長押し3秒。iOSアクセスガイド併用を案内ページで紹介
- カードは2〜6枚・大きく・高コントラスト・アニメーション控えめ

## 4. 支援者ゾーン（編集・履歴・設定）

入口は歯車アイコン。本人モード中は長押し3秒でのみ到達。

**編集**: カードの追加・非表示・並び替え・文言/読み上げ文の変更。絵は内蔵絵ライブラリから選択、**写真カード**はカメラ/アルバムから（端末内のみ保存）。表示形式を**文字だけ／絵＋文字／写真**でテーマ単位切替。問いのON/OFF・逃げ道カードON/OFF・シャッフルON/OFF。カスタムテーマの新規作成も可（同じ仕組みの範囲内）。

**履歴**: 「日時＋テーマ＋選ばれたカード列」を日別一覧（例: `7/16 14:03 きもち → おこってる（とても）`）。**名前は記録しない設計**。任意の「めじるし」欄（1行）を置くが、UI上に**「本名は書かないでください」**を明記。個別削除・全削除・アプリ初期化あり。

**テキスト書き出し**: セッション単位または期間指定で、支援記録に貼れるプレーンテキストをクリップボードへコピー。共有・送信機能は付けない。

## 5. データと安全設計

- すべて端末内: 設定・テーマ・履歴は localStorage、写真は IndexedDB
- サーバー送信なし・アカウントなし・解析なし。通信は初回読み込みと更新確認のみ
- 端末を替えるとデータは引き継がれない（v1の割り切り）
- 本人が写る写真を使う場合の同意の考え方は案内ページに支援者向け注意として記載

## 6. 技術構成

- 置き場所: `/apps/tsutaeru/`（名前は仮）。既存 Astro サイトに同居、main push で本番反映
- 実装: **素の TypeScript のみ**（新規依存なし。サプライチェーン方針と整合）
- PWA: manifest（スコープ `/apps/tsutaeru/`・全画面起動・ホーム画面アイコン）＋自前の小さな Service Worker（一式プリキャッシュ→オフライン起動、接続時に新版取得）
- 読み上げ: Web Speech API（ja-JP）
- **絵ライブラリの自作SVGが最大の制作物**（プリセット全カード＋汎用で60〜80枚。サイト挿絵と同じ温かいトーン・高コントラスト・シンプル）。実装計画で独立工程とする

## 7. 配布導線

案内ページを1枚: アプリ紹介＋QRコード＋「ホーム画面に追加」手順の図解（iPhone: Safari共有→ホーム画面に追加／Android: Chromeのインストール）＋支援者向け注意（同意・個人情報・アクセスガイド）。教えてAIさんの記事と同じトーン。サイトナビからのリンク位置は公開時に相談。

## 8. テスト方針

- Playwright スモーク: テーマ→カード→結果→履歴→書き出しの基本フロー
- ロジック（履歴・書き出し文生成・テーマデータ操作）は純関数に切り出して単体検証
- 実機チェックリスト: iPhone Safari／Android Chrome の「ホーム画面に追加」→**機内モードで起動**、本人モード解除、OS文字サイズ最大、読み上げ

## 9. 却下した代替案

- **ストア配布ネイティブアプリ**: 開発者登録・審査・年会費・保守負担が単機能アプリに見合わない
- **既製AACアプリの紹介のみ**: 理由さぐり・モニタリング・症状伝達という今回の場面設計、事業所カスタム、履歴書き出しが揃わない。案内ページでの併記は可

## 12. 撤退線

- PWAのオフライン挙動が実機で不安定な場合: Service Worker を外し「オンライン専用のホーム画面ショートカット」として一旦公開（機能は全て動く。オフライン対応のみ後日）
- オノマトペカードが現場で通じない場合: カード編集機能で事業所ごとに差し替え（アプリ構造の変更は不要）
