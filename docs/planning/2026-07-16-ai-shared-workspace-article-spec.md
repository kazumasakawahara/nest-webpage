## 2. 設計フェーズ

実装前に、以下を検討して簡潔な設計メモ（このファイルへの追記で可）を作り、**河原さんの承認を得てから実装に進むこと**。

### 2-1. スクリーンショットの内容確認と選定

- 5枚を開き、それぞれ「何の画面か」「記事のどのステップに対応するか」を一覧化する
- 採用枚数は3〜5枚。冗長なら間引いてよい
- **写り込みチェック（必須)**: config.json には filesystem 以外の MCP サーバー設定・ローカルパス・ユーザー名（`k-kawahara`）等が写っている可能性が高い。以下の方針で処理する:
  - **ユーザー名 `k-kawahara` は必ずマスクする（河原さん指示・2026-07-16 確定）。** パスの構造は記事の実例として必要なので、パス自体は残しつつユーザー名部分のみを隠す。方法は「ぼかし」または「`/Users/●●●/...` 等のダミー文字への置き換え合成」のいずれか。全採用画像・全箇所を対象とし、ウィンドウタイトルバー・メニューバー・アカウント表示（メールアドレス等）への写り込みも同様にマスクする
  - filesystem 以外のサーバー設定が写っている場合は、**トリミングまたはぼかしで除外**する（読者を混乱させないためでもある）
  - 万一 PII・認証情報・トークン類が写っていれば**その画像は不採用**とし、代替案を提示

### 2-2. 記事手順節の改稿方針（この方針で仕様化すること）

記事の約束「プログラミングの勉強は要りません」を守りつつ、事実に正直に書く。方向性:

- **2ルート提示**にする:
  - ルートA（できる場合）: 設定画面の「コネクタ」等からフォルダを指定できる場合はそれが最短。※バージョン・プランにより表示が異なる旨を明記
  - ルートB（今回の実機確認ルート・主役）: 設定 → 開発者 → 「構成を編集」で設定ファイルを開き、**AIに教えてもらった数行をコピー＆ペーストする**
- ルートBの筆致: 「設定ファイルを書く」ではなく「**AIが書いてくれたものを、貼り付ける**」に一貫させる。既存の QA 部品（「迷ったら私に聞いてください」）を活かし、「あなたにフォルダをつなぐ設定を書いて」と AI に頼む→返ってきた数行を貼る、という流れにする
- スクリーンショットは figure（`mockup mockup--shot mockup--wide`）としてルートBの各ステップに配置。alt テキストは既存記事同等の粒度（画面に何が写っているかを言葉で説明）で書く
- 「項目名や手順はアプリの更新で変わる」旨の既存注意書きは残す（2026年7月時点の断り書きを追加）
- 他の節（申し送りノートの比喩、使い方3つ、安全上の注意、むすび）は**原則いじらない**。手順変更との整合で最小限の修正のみ可

### 6-2. 【要確認・最重要】仕様書 §0 の前提とスクショの内容が食い違っている

仕様書は「**UIからは設定できず、`claude_desktop_config.json` への直接追記が必要だった。その手順がスクショ5枚に記録されている**」とするが、実際の5枚に **config.json の編集画面は1枚もない**。写っているのは一貫して**コネクタUI経由のルート**である：

> ①アカウントメニュー→設定 → ②設定サイドバー → ③コネクタ→「追加」→「コネクタを参照」 → ④ディレクトリで「file」検索 → ⑤**Filesystem コネクタ詳細（要件充足✓）**

すなわちスクショは、仕様書の言う「ルートA（コネクタUIから設定）」**が現在は可能であること**を示す素材に見える（Filesystem コネクタが公式ディレクトリに存在し、要件も満たしている）。config.json 追記が必要だったのは撮影前の時点の話で、その後 UI ルートが開通した可能性が高い。

また、どちらのルートにせよ**「フォルダを指定する画面」「接続成功（試しに聞いてみる）画面」は5枚に含まれていない**。

### 6-3. 提案（→ 2026-07-16 河原さん回答済み：**案1で承認**。不足スクショ（フォルダ指定・接続確認）は**後日追加撮影・提供**、それまで該当ステップは文章のみ）
